## 1. 문서 요약

### 1.1 서비스 한 줄 정의

국민대학교 학생이 학사·행정·캠퍼스 생활 질문을 한국어로 입력하면, 공식 자료를 근거로 답변하고 출처 확인, 체크리스트, 문의문, 신청서 초안, 졸업 진단까지 이어주는 캠퍼스 생활 액션 에이전트입니다.

### 1.2 핵심 콘셉트

**"국민대 캠퍼스를 탐색하며 학사 행정 퀘스트를 해결하는 근거 기반 액션 에이전트"**

이 서비스는 단순 챗봇이 아닙니다. 학생이 학교 행정 업무를 물어보면, 에이전트가 공식 근거를 찾아 답변하고, 학생이 실제로 다음 행동을 준비할 수 있도록 문서 초안과 체크리스트까지 제공합니다.

### 1.3 서비스의 핵심 가치

| 가치 | 설명 |
| --- | --- |
| 공식 근거 | 국민대학교 공식 공개 자료 chunk와 citation 기반 답변 |
| 개인정보 보호 | 학번, 주민번호, 연락처, 포털 ID/PW, 성적표 원본 입력 차단 |
| 다음 행동 | 답변 후 신청서 초안, 문의문, 체크리스트로 연결 |
| 학생 언어 이해 | 이캠, 과사, 국장, 종정시 같은 학생식 표현 이해 |
| 캠퍼스형 UX | 건물, NPC, 퀘스트, 서류함, 대백과, 졸업센터로 구성 |
| 운영 검증성 | tool log, source panel, health, ingest 상태 확인 가능 |

### 1.4 현재 구현 상태 요약

- 프론트엔드는 Vite + React 기반 단일 페이지 앱입니다.
- 백엔드는 FastAPI 기반 API 서버입니다.
- 일반 답변은 deterministic RAG orchestration으로 생성됩니다.
- Chroma vector 검색은 optional accelerator이며 keyword 검색이 fallback입니다.
- 공식 자료 수집은 public source만 대상으로 하며 post-login 화면은 수집하지 않습니다.
- 답변에는 `[S1]`, `[S2]` citation marker와 출처 목록이 포함됩니다.
- 후속 action은 `/actions/start`, `/actions/continue` 상태 머신으로 동작합니다.
- 졸업센터는 별도 API와 비식별 transcript summary 기반 분석 흐름을 가집니다.

## 2. 문제 정의

### 2.1 학생이 겪는 문제

국민대학교 학생은 학사·행정 정보를 찾을 때 여러 공식 페이지를 오가야 합니다.

예시:

- 학사안내
- 학사일정
- 학사공지
- 행정공지
- 장학공지
- 학생지원 안내
- ON국민 포털
- eCampus
- 생활관, 도서관, 식당, 통학버스 등 별도 사이트

이 과정에서 학생은 아래 어려움을 겪습니다.

| 문제 | 상세 |
| --- | --- |
| 정보 분산 | 같은 행정 절차가 학사안내, 공지, 일정, 포털에 나뉘어 있음 |
| 용어 장벽 | 공식 용어와 학생이 쓰는 표현이 다름 |
| 절차 불확실성 | 어디에, 언제까지, 어떤 서류로 진행해야 하는지 불명확 |
| 담당 부서 혼동 | 교무팀, 학과사무실, 종합서비스센터, 학생지원팀 구분이 어려움 |
| 개인정보 불안 | 학번, 성적, 연락처를 입력해야 하는지 헷갈림 |
| 답변 신뢰 부족 | 일반 챗봇 답변은 근거와 최신성을 확인하기 어려움 |
| 후속 작업 부담 | 답변을 읽은 뒤 문의문, 신청서, 체크리스트를 따로 작성해야 함 |

### 2.2 기존 검색/챗봇 방식의 한계

일반 검색은 URL과 문서 목록을 보여주지만 학생이 직접 해석해야 합니다.

일반 챗봇은 설명을 잘할 수 있지만, 아래 문제가 있습니다.

- 공식 근거 없이 행정 절차를 단정할 수 있습니다.
- 최신 학사일정과 공지를 놓칠 수 있습니다.
- 개인정보 입력을 유도할 위험이 있습니다.
- 답변 이후 실제 제출/문의 준비까지 연결되지 않습니다.

### 2.3 본 서비스의 접근

이 서비스는 학생 질문을 아래 순서로 처리합니다.

```text
학생 질문
→ 개인정보 차단
→ 이슈 분류
→ 공식 자료 검색
→ citation 기반 답변
→ 체크리스트/문의처/기한 생성
→ 다음 행동 추천
→ 문서 초안 또는 졸업 진단 연결
```

핵심은 "검색 결과 제공"이 아니라 "안전하고 근거 있는 다음 행동 준비"입니다.

## 4. 서비스 포지셔닝

### 4.1 제품 정체성

이 서비스는 아래 세 제품 유형의 교차점에 있습니다.

| 유형 | 서비스 내 역할 |
| --- | --- |
| RAG 챗봇 | 공식 자료 기반 Q&A |
| 행정 액션 도우미 | 신청서, 문의문, 체크리스트 초안 |
| 캠퍼스 탐색 UI | 건물/NPC/퀘스트 기반 질문 진입 |

### 4.2 하지 않는 것

| 하지 않는 일 | 이유 |
| --- | --- |
| 실제 행정 신청 제출 대행 | 권한과 책임 범위 초과 |
| ON국민/eCampus 로그인 대행 | 포털 ID/PW 수집 금지 |
| 개인 화면 조회 | post-login 개인정보 접근 금지 |
| 성적표 원본 저장 | 민감 학업정보 보호 |
| 공식 근거 없는 절차 조언 | hallucination 방지 |
| 최종 졸업 판정 | 학과/교무팀 권한 |
| 에브리타임 등 비공식 커뮤니티 수집 | 출처 신뢰도 문제 |

### 4.3 서비스 원칙

1. 개인정보를 받지 않습니다.
2. 공식 근거가 없으면 모른다고 말합니다.
3. 절차성 답변에는 citation을 붙입니다.
4. 학생이 바로 실행할 수 있는 단위로 안내합니다.
5. 최종 확인 책임은 공식 포털, 담당 부서, 학과사무실, 담당 교강사에 둡니다.

## 5. 서비스 구조

### 5.1 전체 구조

```text
사용자
  → React 프론트엔드
      ├─ 캠퍼스 맵
      ├─ 질문/대화 UI
      ├─ 출처 패널
      ├─ 서류함
      ├─ 졸업센터
      └─ 연구소/관리자 패널
  → FastAPI 백엔드
      ├─ Privacy Guard
      ├─ Issue Classifier
      ├─ Hybrid Retriever
      ├─ Action Planner
      ├─ Answer Builder
      ├─ Document Drafter
      ├─ Ingestion Pipeline
      └─ Graduation Center
  → 공식 공개 자료 / JSONL / Chroma / 졸업 구조화 데이터
```

### 5.2 프론트엔드 위치

프론트엔드는 서비스의 경험 레이어입니다.

주요 책임:

- 캠퍼스형 진입 경험 제공
- 학생 상태와 관심 항목 입력
- 질문 입력과 답변 표시
- citation과 source panel 연결
- action form 렌더링
- 졸업센터 모달 제공
- live check, GPT 보조, tool logs 상태 표시

### 5.3 백엔드 위치

백엔드는 서비스의 정책·검색·생성 레이어입니다.

주요 책임:

- 개인정보 guard
- 질문 분류
- 공식 자료 검색
- source guard
- 답변 조립
- action 상태 관리
- 공식 자료 수집/갱신
- 졸업센터 분석

### 5.4 에이전트 위치

에이전트는 서비스의 판단 orchestration 레이어입니다.

주요 책임:

- 어떤 이슈인지 판단
- 어떤 근거를 사용할지 선택
- 어떤 action을 추천할지 결정
- 공식 근거와 학생 경험 팁을 구분
- LLM을 사용하더라도 deterministic contract를 유지

## 7. 핵심 사용자 여정

### 7.1 여정 A: 빠른 질문에서 근거 기반 답변까지

```text
서비스 접속
→ 질문 입력 또는 추천 질문 선택
→ 개인정보 guard
→ issue_type 분류
→ 공식 자료 검색
→ 답변 조립
→ citation 포함 답변 표시
→ 출처 패널 확인
→ 다음 행동 선택
```

성공 기준:

- 사용자는 답변이 어떤 공식 자료에 근거하는지 확인할 수 있습니다.
- 사용자는 해야 할 일, 필요서류, 신청 경로, 문의처를 한 화면에서 파악할 수 있습니다.

### 7.2 여정 B: 건물 탐색에서 질문까지

```text
캠퍼스 맵 확인
→ 건물 선택
→ NPC/추천 질문 확인
→ 질문 선택
→ 답변 확인
→ 퀘스트 완료 또는 action 연결
```

성공 기준:

- 사용자는 정확한 행정 용어를 몰라도 건물과 상황을 통해 질문에 진입할 수 있습니다.
- 퀘스트 방식으로 행정 업무의 심리적 진입 장벽을 낮춥니다.

### 7.3 여정 C: 답변에서 문서 초안까지

```text
답변 하단 next action 선택
→ /actions/start
→ 필요한 slot 질문 표시
→ 개인정보 없는 값 입력
→ /actions/continue
→ 문서 초안과 체크리스트 반환
→ 사용자가 공식 양식/담당자 안내에 맞게 최종 수정
```

성공 기준:

- 사용자는 빈 문서 앞에서 막히지 않습니다.
- 초안은 개인정보 없이 작성됩니다.
- 최종 제출 전 확인해야 할 항목이 분리됩니다.

### 7.4 여정 D: 복학생 일정 확인

```text
학적 카드에서 복학생 선택
→ 대상 학기와 관심 항목 입력
→ "이번 주 뭐 해야 해?" 질문
→ schedule issue 분류
→ 학사일정 검색
→ 복학생 맞춤 확인 표시
```

성공 기준:

- 복학 승인, 수강신청 가능 상태, 등록금 고지서 반영처럼 복학생 특화 항목이 표시됩니다.
- 개인 신청 상태는 ON국민에서 직접 확인해야 한다는 안내가 포함됩니다.

### 7.5 여정 E: 졸업센터 분석

```text
졸업센터 진입
→ 개인정보 안내 확인
→ 비식별 transcript summary 또는 PDF parse 진행
→ 졸업/대체과목/진로/조기졸업 task 선택
→ 분석 결과 확인
→ 학과/교무팀 확인 질문 정리
```

성공 기준:

- 원본 성적표 저장 없이 분석이 이뤄집니다.
- 최종 졸업 판정이 아니라 확인 항목과 다음 행동으로 제공됩니다.

- 포털 ID/PW

- 성적표 원본

기획 UX:

- 차단 시 대체 입력 방식을 안내합니다.

### 8.6 졸업센터

졸업센터는 일반 Q&A보다 깊은 분석 기능입니다.

주요 기능:

- 성적증명서 PDF parse
- 졸업요건 audit
- 대체과목 확인
- 마이크로디그리 분석
- 졸업 후 행정 체크리스트
- 직무 역량 번역
- 조기졸업 가능성 확인
- 자기설계/Customized 전공 안내
- 학점포기/성적 관련 정책 확인

핵심 제약:

- 성적표 원본을 저장하지 않습니다.
- 비식별 summary를 중심으로 분석합니다.
- 최종 졸업 판정은 학교 담당 부서 확인이 필요합니다.

### 8.7 관리자/운영 기능

| 기능 | 목적 |
| --- | --- |
| `/health` | keyword/vector/LLM/graduation/ingest 상태 확인 |
| `/sources` | indexed official source 목록 확인 |
| `/ingest/run` | 공식 자료 수집/인덱스 갱신 |
| `/ingest/live-refresh` | issue별 짧은 최신 확인 |
| tool logs | 처리 단계와 fallback 여부 확인 |

운영 주의:

- 운영 환경에서는 ingest/admin API에 인증이 필요합니다.
- CORS 정책은 운영 도메인 기준으로 제한해야 합니다.
- 학교 서버 보호 정책을 완화하면 안 됩니다.

## 9. 데이터와 출처 정책

### 9.1 공식 출처 원칙

사용 가능한 출처는 국민대학교 공식 공개 자료입니다.

우선순위:

| source_tier | 출처 | 의미 |
| --- | --- | --- |
| 1 | 규정관리시스템 | 제도적 최상위 근거 |
| 2 | 학사안내 | 절차, 서류, 신청경로 |
| 3 | 학생지원/대학생활 안내 | 학생증, 증명서, 병무, 상담 |
| 4 | 학사일정 | 기간, 마감일 |
| 5 | 공지사항 | 학기별 변경/모집/신청 공지 |
| 6 | 요람/규정집 | 졸업요건, 교육과정 |
| 7 | 대학조직 | 문의처/부서 |
| 8 | SWELL 공개 게시판 | 공개 비교과/신청 안내 |

### 9.2 수집 정책

- 공개 페이지에 한정합니다.
- 로그인 후 개인 화면은 수집하지 않습니다.
- 에브리타임 같은 비공식 커뮤니티는 수집하지 않습니다.
- per-host delay를 지킵니다.
- run별 최대 page 수를 제한합니다.
- cooldown과 ingest lock을 유지합니다.
- 네트워크 실패 시 curated fallback을 사용하되 상태를 응답에 표시합니다.

### 9.3 chunk metadata

서비스 기능은 chunk metadata에 의존합니다.

| metadata | 쓰임 |
| --- | --- |
| `issue_types` | 질문 이슈와 source matching |
| `keywords`, `search_hints` | 검색 품질 개선 |
| `required_documents` | 필요 서류 표시 |
| `application_path` | 신청 경로 표시 |
| `submit_to` | 제출처 표시 |
| `contacts` | 문의처 추천 |
| `schedule` | 일정과 마감 표시 |
| `deadline_rule` | 기한 계산 |
| `actions` | 다음 행동 추천 |
| `used_fallback`, `fetch_status` | 최신성/수집 상태 표시 |

## 11. 사용자 경험 원칙

### 11.1 화면 원칙

- 첫 화면은 서비스 설명보다 실제 사용 가능한 캠퍼스/질문 경험이어야 합니다.
- 건물/NPC/퀘스트는 질문 진입을 돕는 장치로 사용합니다.
- 답변과 출처, 다음 행동은 한 흐름 안에서 연결되어야 합니다.
- 관리자/디버그 정보는 학생용 흐름을 방해하지 않게 분리합니다.
- 모바일에서는 채팅, 출처, 서류함 패널이 겹치지 않아야 합니다.

### 11.2 문장 원칙

좋은 문장:

- "공식 근거에서 확인되는 내용은 다음과 같습니다."
- "개인 신청 상태는 ON국민 포털에서 직접 확인해야 합니다."
- "문의 전에는 아래 정보를 개인정보 없이 정리해 주세요."
- "최종 처리는 담당 부서 확인이 필요합니다."

피해야 할 문장:

- "승인됩니다."
- "확정입니다."
- "학번을 알려주세요."
- "비밀번호를 입력하세요."
- "제가 포털에서 확인했습니다."

### 11.3 개인정보 UX

- 입력창 주변에 개인정보 입력 금지 안내를 둡니다.
- action form에도 동일한 안내를 유지합니다.
- 차단 시 사용자가 다시 시도할 수 있는 안전한 입력 예시를 제공합니다.
- 졸업센터는 업로드 전 privacy notice와 OCR consent를 명확히 보여줍니다.

## 12. 대표 시나리오

### 12.1 예비군 출석인정

질문:

```text
예비군 훈련 때문에 수업 빠지면 출석인정 어떻게 해?
```

기대 경험:

- issue_type은 `attendance`
- 출석인정 관련 공식 근거와 예비군 관련 근거 검색
- 증빙서류와 제출기한 확인
- 담당 교강사/교무팀 문의처 표시
- `출석인정신청서 초안 작성` action 추천

### 12.2 eCampus 강의 미표시

질문:

```text
이캠에 강의가 안 떠요
```

기대 경험:

- issue_type은 `portal_access`
- 학생식 표현 `이캠`을 eCampus로 해석
- 수강신청 완료 여부와 강의 공개 시점 확인 안내
- 비밀번호/학번 입력 금지 안내
- `포털/eCampus 접근 체크리스트`와 문의문 action 추천

### 12.3 복학생 이번 주 할 일

질문:

```text
복학생인데 이번 주 뭐 해야 해?
```

기대 경험:

- issue_type은 `schedule`
- 복학생 맞춤 확인 섹션 표시
- 복학 승인, 수강신청 가능 상태, 등록금 고지서 반영 여부 안내
- 진행 중/다가오는 학사일정 표시
- 개인 상태는 ON국민에서 직접 확인 안내

### 12.4 등록금 납부확인

질문:

```text
등록금 냈는데 납부확인이 안 떠요
```

기대 경험:

- issue_type은 `registration_tuition`
- 은행 납부와 포털 반영 시간을 분리해서 설명
- 영수증, 납부 일시, 납부 방식 준비 안내
- 재무팀/등록 담당 부서 문의처 추천
- 개인정보 없는 문의문 초안 action 추천

### 12.5 졸업요건 확인

질문:

```text
졸업요건 부족한지 알고 싶어요
```

기대 경험:

- issue_type은 `graduation`
- 요람/학과 기준 확인 필요 안내
- 성적표 원본 입력 금지 안내
- 졸업요건 간이 진단 action 또는 졸업센터 연결
- 최종 판정은 학과사무실/교무팀 확인 안내

## 13. 운영과 품질 관리

### 13.1 서비스 상태 확인

운영자는 `/health`에서 아래 상태를 확인합니다.

- keyword chunk 수
- vector retriever availability
- vector indexed count
- vector error
- LLM 상태
- 졸업센터 상태
- 마지막 ingest 상태
- live refresh 최근 결과

### 13.2 데이터 품질 관리

확인 항목:

- 공식 URL이 유효한가
- source_tier가 적절한가
- issue_types가 정확한가
- required_documents/application_path/contacts가 구조화되어 있는가
- schedule 날짜가 ISO format인가
- fallback_text가 공식 안내의 핵심을 반영하는가

### 13.3 답변 품질 관리

확인 항목:

- 질문 의도와 issue_type이 맞는가
- 검색 chunk가 직접 관련되는가
- 절차성 주장에 citation이 있는가
- 필요서류, 신청경로, 기한, 문의처가 가능한 범위에서 표시되는가
- 개인정보 입력을 요구하지 않는가
- 최종 확인 필요 문구가 포함되는가

### 13.4 액션 품질 관리

확인 항목:

- slot 질문이 최소한인가
- 개인정보를 요구하지 않는가
- 초안이 바로 수정 가능한 형태인가
- 체크리스트가 실행 순서로 정리되어 있는가
- 공식 양식/담당자 확인 필요 문구가 포함되는가

## 15. 릴리즈 기준

### 15.1 MVP 릴리즈 기준

- `/ask`가 주요 issue_type에 대해 정상 답변합니다.
- 개인정보 입력이 차단됩니다.
- 공식 근거가 없으면 절차 답변을 생성하지 않습니다.
- citation marker와 source panel이 연결됩니다.
- 최소 5개 이상 action이 정상 완료됩니다.
- 프론트에서 질문, 출처, action, 관리자 상태를 확인할 수 있습니다.
- 졸업센터는 privacy notice와 함께 제한 기능으로 제공됩니다.

### 15.2 데모 릴리즈 기준

- 프론트 `5173`, 백엔드 `8001` 실행 절차가 명확합니다.
- 추천 질문 6개 이상이 성공적으로 답변됩니다.
- 예비군 출석인정, 이캠 강의 미표시, 복학생 일정, 등록금 납부확인, 졸업요건 시나리오가 통과합니다.
- 서류함에서 출석인정신청서 또는 문의문 초안이 생성됩니다.
- `/health`에서 서비스 상태를 설명할 수 있습니다.

### 15.3 운영 릴리즈 전 필요 항목

- 관리자 API 인증 추가
- CORS 운영 도메인 제한
- ingest 실행 권한 분리
- 개인정보 처리방침/안내문 정리
- 성적증명서 업로드 처리 정책 명문화
- 로그 보관 범위와 민감정보 마스킹 정책 확정
- 공식 자료 수집 주기와 실패 대응 프로세스 정의

## 19. 최종 정리

KMU 캠퍼스 생활 액션 에이전트는 국민대학교 학생의 행정 질문을 "공식 근거가 있는 답변"과 "실행 가능한 다음 행동"으로 바꾸는 서비스입니다.

프론트엔드는 캠퍼스 탐색과 퀘스트로 접근성을 만들고, 백엔드는 공식 자료 검색과 안전 정책을 책임지며, 에이전트는 질문 이해와 action 연결을 orchestration합니다.

서비스가 지켜야 할 가장 중요한 기준은 아래와 같습니다.

1. 개인정보를 요구하지 않는다.
2. 공식 근거 없는 절차를 만들지 않는다.
3. citation으로 답변을 검증 가능하게 한다.
4. 학생이 다음 행동을 준비할 수 있게 한다.
5. 최종 행정 판단은 학교 공식 경로로 연결한다.

이 기준이 유지될 때, 서비스는 단순 FAQ나 챗봇을 넘어 학생의 실제 캠퍼스 생활 업무를 줄여주는 안전한 학사 행정 도우미가 됩니다.
